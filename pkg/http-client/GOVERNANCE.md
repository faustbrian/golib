# Governance

The project uses a maintainer-led model. Brian Faust is the initial maintainer
and release authority. Maintainers decide roadmap, merge, release, and security
matters after considering public technical evidence and compatibility impact.

Routine changes require one maintainer approval and passing required checks.
Changes to exported APIs, defaults, retry or redirect behavior, security
boundaries, error contracts, middleware ordering, telemetry labels, or fixture
schemas require an explicit compatibility note. Security releases may be
prepared privately until coordinated disclosure.

New maintainers are added based on sustained, trustworthy contributions and
sound review judgment. Inactive maintainers may become emeritus. Governance
changes are reviewed like other public contract changes and recorded in the
changelog.
