# Goal: Production-Grade Three-Dimensional Bin Packing for Go

## Normative Language

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT",
"SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and
"OPTIONAL" in this document are to be interpreted as described in BCP 14
[RFC2119] [RFC8174] when, and only when, they appear in all capitals, as
shown here.

## Objective

Build `knapsack` as a production-grade open-source Go library for packing
rectangular physical items into one or more rectangular containers while
respecting dimensions, orientation, weight, stock, stability, load-bearing, and
application-defined constraints.

The module path MUST be `github.com/faustbrian/golib/pkg/knapsack`. Public Go package
identifiers MUST use `knapsack` or focused non-stuttering subpackage names.
Go 1.26 MUST be the minimum supported language and toolchain version at initial
release, updated only through the repository-wide version policy.

The primary e-commerce problem is:

> Given item instances and available box types, produce a feasible placement
> for every item while minimizing an explicit objective such as box count or
> total packaging cost.

This problem is commonly called “4D” bin packing because it combines three
physical dimensions with weight. The documentation MUST explain that weight is
a scalar capacity constraint, not a fourth geometric axis. The library MUST
distinguish three-dimensional bin packing from classical value-maximizing
knapsack selection and MUST NOT make unsupported optimality claims.

## Design Principles

- Correct feasibility outranks packing density or speed.
- Every result MUST be independently verifiable.
- Determinism MUST be the default.
- Units, precision, objectives, constraints, and search budgets MUST be explicit.
- A heuristic result MUST not be described as optimal without proof.
- Search MUST be bounded and cancellable.
- Domain objects and results MUST be immutable from the caller's perspective.
- The root package MUST remain usable without services, databases, queues,
  telemetry systems, or framework runtime dependencies.
- Advanced algorithms MUST remain additive rather than forcing every consumer
  to pay their dependency or complexity cost.

## Problem Definitions

The package MUST support these separate operations:

1. **Pack all items:** choose container instances and place every item while
   minimizing an explicit objective.
2. **Pack into fixed containers:** place as many or all items into an already
   supplied finite container set without inventing additional containers.
3. **Select a subset:** optionally solve the classical constrained-knapsack
   variant by selecting item values/priorities under geometric and weight
   constraints. This API MUST remain distinct from pack-all behavior.
4. **Verify a supplied plan:** validate a placement generated by this or another
   solver without rerunning optimization.

Pack-all is Phase 1 and MUST be complete before subset selection is advertised.
The first release MUST target offline orthogonal cuboid packing. Online packing,
irregular meshes, cylinders as true curved geometry, pallet loading, vehicle
axle loads, hazardous-goods regulation, and robotic motion planning are outside
the initial scope unless added through separately specified packages.

## Numeric And Unit Model

Physical quantities MUST be exact and unit-safe:

- lengths and mass MUST use `measurement` values at public boundaries;
- conversion MUST use exact `math` arithmetic;
- a packing request MUST choose an explicit positive integer lattice or exact
  resolution before search begins;
- conversion to the lattice MUST reject inexact values unless the caller
  selects and documents an explicit rounding policy;
- dimensions, coordinates, volumes, weights, costs, scores, and intermediate
  products MUST be checked for overflow before allocation or arithmetic;
- `float32` and `float64` MUST NOT be the authoritative feasibility model; and
- mixed units MUST never be interpreted as already normalized.

The package MUST NOT invent duplicate decimal, measurement, or money types.
If keeping monetary optimization dependency-additive requires a nested adapter
module, `objective/gomoney` MUST bridge `money` without weakening exact
comparison or introducing a second public cost representation.

The API MUST make physical measurement tolerance distinct from algorithmic
search tolerance. Silent epsilon-based overlap or boundary acceptance is
forbidden.

## Core Domain Model

### Items

An item instance MUST have:

- a stable non-empty instance ID;
- original dimensions on three named axes;
- weight;
- an explicit set of allowed orthogonal orientations;
- optional type/SKU metadata that does not participate implicitly in packing;
  and
- optional priority/value only when used by an objective that defines it.

The model SHOULD support:

- quantity expansion with deterministic unique instance identity;
- upright-only, keep-flat, and caller-defined orientation restrictions;
- fragile or non-stackable top surfaces;
- maximum supported weight and load-bearing behavior;
- required minimum support ratio;
- grouping/linked-item preferences or requirements;
- separation/incompatibility constraints;
- maximum stack count or stack height;
- packing priority without silently dropping lower-priority items; and
- custom typed attributes consumed only by explicit constraints/objectives.

Negative, zero, non-finite, inexact, duplicate-ID, overflowed, or otherwise
invalid item data MUST be rejected before solving.

### Containers

A container type MUST have:

- a stable non-empty type ID;
- usable internal dimensions distinct from optional external dimensions;
- maximum content weight and optional tare weight/max gross weight;
- finite stock or explicitly unlimited availability;
- optional monetary cost represented through `money` only when cost
  optimization is requested; and
- deterministic priority/tie-break metadata.

The model SHOULD support maximum item count, reserved rectangular keep-out
regions, allowed item classes, container-specific constraints, and optional
center-of-gravity bounds. Non-rectangular containers MUST NOT be approximated as
fully usable cuboids without an explicit caller-provided policy.

### Placements And Plans

Every placement MUST expose:

- item instance ID and container instance ID;
- integer-lattice origin `x`, `y`, and `z`;
- selected orientation;
- oriented dimensions and weight;
- supporting surface/item relationships where stability is enabled; and
- any objective or constraint diagnostics needed to explain the decision.

A packing plan MUST expose:

- immutable container instances and placements;
- packed and unpacked item instances without loss or duplication;
- total item/container weight and volume accounting;
- used and remaining capacity;
- objective definition and ordered score components;
- solver, strategy, seed if any, limits, elapsed work, and termination reason;
- feasibility-verification result;
- proof status: `feasible`, `optimal`, `best_known`, `infeasible`,
  `budget_exhausted`, or another precisely documented state; and
- structured diagnostics for every unpacked or impossible item.

`infeasible` MUST mean infeasibility was proven. Failure by a heuristic or
search-budget exhaustion MUST use a different status.

## Feasibility Invariants

The independent verifier MUST prove all enabled invariants:

- every packed item appears exactly once;
- every required item is packed unless the operation permits unpacked items;
- every placement lies completely inside usable container space;
- no two occupied item cuboids overlap;
- no item overlaps a reserved region;
- selected orientation is allowed;
- container content/gross weight is not exceeded;
- finite container stock is not exceeded;
- support ratio, load-bearing, fragility, and stack limits hold;
- grouping and incompatibility constraints hold;
- all coordinates and totals use the declared lattice and units; and
- recomputed statistics exactly match the reported result.

The verifier MUST be structurally independent from placement-generation logic
so the same defect cannot construct and approve an invalid plan.

## Orientation And Geometry

The geometry package MUST:

- enumerate the six axis-aligned cuboid orientations and deduplicate equivalent
  rotations for repeated dimensions;
- represent half-open occupied intervals consistently;
- provide exact containment, intersection, adjacency, support-area, free-space,
  and volume operations;
- define coordinate origin and axis conventions publicly;
- reject overflow and invalid boxes before geometric operations; and
- remain deterministic regardless of map iteration or input ordering.

Rotation restrictions MUST operate on physical axes, not incidental enum order.
Touching faces/edges are allowed; positive-volume overlap is forbidden.

## Objectives And Tie Breaking

The package MUST provide explicit lexicographic objectives rather than one
opaque weighted score. Built-in objectives SHOULD include:

- minimize container count;
- minimize total packaging cost;
- minimize unused volume after the primary objective;
- minimize unused weight capacity;
- balance content weight across containers;
- minimize maximum used height or another documented loading metric; and
- maximize packed priority/value for subset-selection operations.

Every objective MUST define direction, units, comparison, overflow behavior,
and deterministic final tie-break. Caller-defined objectives MUST be pure,
bounded, panic-safe at the API boundary, and unable to mutate solver state.

Container count and cost are not interchangeable. A cheaper multi-box plan may
use more boxes; the caller MUST choose the precedence explicitly.

## Solvers And Search Budgets

Phase 1 MUST include a deterministic production heuristic suitable for ordinary
e-commerce orders and a bounded exact solver for small instances.

The heuristic SHOULD combine independently testable components:

- canonical item sortation;
- candidate container selection;
- extreme-point, pivot-point, maximal-space, level, or other documented
  placement generation;
- placement scoring;
- cross-container improvement/repacking; and
- deterministic final tie-breaking.

The exact solver MUST be usable as a small-instance oracle and MUST report
optimality only after exhausting or correctly pruning the complete relevant
search space. Branch-and-bound lower bounds and symmetry elimination MUST be
documented and tested against brute force.

Every solver MUST accept `context.Context` and explicit limits for:

- item and container-type count;
- expanded quantity;
- orientations and candidate placements;
- search nodes, branches, permutations, and improvement rounds;
- wall-clock deadline and cancellation;
- memory and retained-result count;
- goroutines/parallelism; and
- deterministic random seed for any stochastic strategy.

Cancellation and budget exhaustion MUST return the best independently verified
plan found so far when safe, plus an unambiguous termination reason. Solvers MUST
not leak goroutines or continue consuming resources after return.

## Constraints And Extension Model

Provide small, typed extension interfaces for feasibility constraints,
placement scoring, item ordering, container ordering, and complete-plan
objectives. Interfaces MUST be segregated by actual consumer need and MUST NOT
expose mutable solver internals.

Custom constraints MUST receive bounded immutable views and return structured
accept/reject diagnostics. Panics from callbacks MUST either be converted to a
documented error or explicitly documented as caller failure; behavior MUST be
consistent across sequential and parallel solvers.

Built-in constraints MUST not be implemented through a slower generic callback
path where direct verified geometry is required for correctness or performance.

## Determinism And Reproducibility

For identical normalized input, options, solver version, and seed, the result
MUST be byte-for-byte canonically serializable and independent of:

- input slice permutation when item/container IDs and objective semantics are
  unchanged;
- Go map iteration;
- goroutine scheduling;
- CPU count; and
- wall-clock timing before an explicit budget is exhausted.

When multiple plans are equally scored, documented canonical ID, orientation,
container, and coordinate tie-breaks MUST choose one. Parallel optimization MAY
find candidates concurrently but MUST reduce them deterministically.

## Errors And Diagnostics

Public errors MUST support `errors.Is`/`errors.As` and distinguish:

- invalid item/container/request/options;
- unit conversion or lattice-resolution failure;
- impossible item dimensions;
- item overweight for every eligible container;
- insufficient finite stock;
- conflicting constraints;
- no feasible placement found by the selected strategy;
- proven infeasibility;
- cancellation, deadline, node/memory/work budget exhaustion; and
- internal invariant/verifier failure.

Diagnostics SHOULD explain rejected orientations and candidate container types
without exposing unbounded internal search traces. Error strings are for humans;
machine behavior MUST use stable typed categories and fields.

## Serialization And Visualization

Provide deterministic JSON encoding for normalized requests, plans,
placements, diagnostics, and benchmark fixtures. Decoding MUST be strict,
versioned, bounded, reject duplicate keys where ambiguity matters, preserve
exact units, and validate before solving.

An additive visualization package MAY render SVG or a neutral scene description
for debugging. Visualization MUST consume a verified plan and MUST not become a
required root dependency. A visual rendering is not feasibility proof.

## Package Structure

Prefer a focused layout such as:

```text
knapsack/
  knapsack.go            small top-level API
  item.go
  container.go
  plan.go
  options.go
  errors.go
  geometry/              exact cuboid and support operations
  constraint/            built-in constraints and extension contracts
  objective/             lexicographic objectives
  objective/gomoney/     optional money cost objective adapter
  solver/                bounded heuristic and exact solvers
  verify/                independent plan validator
  encoding/              strict versioned JSON adapters
  visualize/             optional diagnostics rendering
  knapsacktest/          fixtures, assertions, and verifier helpers
```

Package boundaries MUST follow real dependency direction. Do not create an
interface, package, or generic abstraction merely to imitate another language
or inflate architecture.

## Interoperability And Reference Implementations

The project MUST inventory, pin, and license-review relevant references,
including:

- `dvdoug/BoxPacker` for the established e-commerce behavior set;
- `jcoruiz/gopackx` for current Go variable-box and physical-constraint behavior;
- `gedex/bp3d` for the common older Go API/heuristic baseline; and
- independently published exact solvers or benchmark corpora suitable for
  small-instance optimality checks.

BoxPacker documents rotation, sortation, weight distribution, too-large items,
positional information, limited box supply, custom constraints, linked items,
and used/remaining space. Phase 1 MUST assess each capability and classify it as
implemented, intentionally different, deferred with rationale, or out of scope.

Reference code MUST NOT be copied without license and provenance review.
Self-generated fixtures and self-round trips are not independent evidence.

## Testing Standard

Meaningful 100% production statement coverage is REQUIRED. It MUST be backed by
tests that detect realistic defects rather than merely execute lines.

Required suites include:

- table-driven unit tests for every geometry, orientation, constraint,
  objective, status, error, and tie-break branch;
- property and metamorphic tests for containment, non-overlap, permutation,
  scaling, symmetry, monotonic capacity, and conservation invariants;
- independent verification of every solver result in every test and benchmark;
- brute-force/exact comparison for exhaustively enumerable small cases;
- recognized benchmark corpora with pinned provenance and expected bounds;
- differential tests against reference implementations on the exact common
  semantic subset;
- golden tests for canonical requests, placements, diagnostics, and encoding;
- fuzzing of request decoding, normalization, geometry, constraints, solver
  entry points, verification, and malformed supplied plans;
- race and deterministic-parallel tests;
- mutation testing of all feasibility predicates, bounds, pruning, orientation,
  scoring, status, and accounting logic;
- cancellation, deadline, memory/work limit, and goroutine-leak tests; and
- architecture, API compatibility, examples, and documentation tests.

Coverage exclusions MUST be narrow, generated-only, and reviewed. Every
eligible meaningful mutant MUST be killed or classified with proof of semantic
equivalence/tool limitation.

## Benchmarking Standard

Benchmarks MUST measure both speed and solution quality. Record:

- items and container types;
- constraints and objective;
- feasibility and verification time;
- objective score, box count, cost, utilization, and optimality gap where known;
- solver work/nodes/iterations;
- wall time, allocations, bytes, peak memory, and parallelism; and
- machine, Go version, commit, seed, and fixture hash.

Maintain suites for tiny exact, ordinary e-commerce, orientation-heavy,
weight-limited, stability/load-bearing, finite-stock, adversarial, impossible,
and production-scale cases.

Comparisons with BoxPacker, GoPackX, `bp3d`, or another solver MUST use identical
normalized dimensions, weights, rotations, stock, constraints, and objectives.
Unsupported semantics MUST be removed from both sides or reported separately.
Do not compare only runtime when one solver returns a worse or invalid plan, and
do not advertise “faster”, “better”, or “optimal” without reproducible evidence.

## Security And Resource Safety

- Validate all counts, quantities, dimensions, weights, costs, IDs, and limits
  before expansion or allocation.
- Bound NP-hard search work before accepting untrusted requests.
- Prevent integer overflow in volume, coordinates, products, scores, indexes,
  capacities, and allocation sizes.
- Reject duplicate IDs and ambiguous serialized input.
- Ensure custom callbacks cannot mutate aliased package state.
- Treat serialized requests/plans and visualization labels as untrusted.
- Never execute expressions or load plugins from input.
- Return bounded diagnostics; never emit an unbounded search trace by default.
- Honor cancellation promptly and prove no panic, deadlock, livelock, or leak
  under hostile input.

`SECURITY.md`, `CONTRIBUTING.md`, `CHANGELOG.md`, code of conduct, support and
governance policies, examples, and package documentation. Changelog maintenance

## Implementation Phases

1. **Foundation:** repository setup, exact unit/lattice model, domain types,
   geometry, errors, strict options, and independent verifier.
2. **Feasible packing:** deterministic fixed-container placement with complete
   invariants and diagnostics.
3. **Variable boxes:** pack-all container selection, finite stock, weight,
   explicit objectives, and cross-container improvement.
4. **Physical constraints:** orientation policies, support, fragility,
   load-bearing, grouping, separation, and keep-out regions.
5. **Exact oracle:** bounded small-instance exact solver, proof status, lower
   bounds, and brute-force verification.
6. **Interoperability:** BoxPacker capability matrix, Go differential tests,
   benchmark corpora, canonical encoding, and optional visualization.
7. **Hardening:** meaningful 100% coverage, mutation, fuzz, race, cancellation,
   leak, hostile-input, performance, quality, docs, and release gates.
8. **Optional subset selection:** classical value-maximizing knapsack only after
   the pack-all API and terminology are stable.

## Release Blockers

- Any plan accepted with overlap, boundary escape, overweight container,
  forbidden orientation, stock excess, unsupported load, or accounting drift.
- Item loss, duplication, or identity ambiguity.
- A heuristic failure labeled `infeasible` or a non-proven result labeled
  `optimal`.
- Float-dependent feasibility, silent unit conversion, hidden rounding, or
  arithmetic overflow.
- Nondeterministic output without an explicit seeded mode.
- Unbounded search, quantity expansion, allocation, diagnostics, parallelism,
  callback work, or retained state.
- Ignored cancellation, panic, race, deadlock, livelock, or goroutine leak.
- Self-verification using only the same placement predicates as the solver.
- Dubious benchmark comparisons or quality claims without equivalent semantics.
- Missing reference/corpus provenance or incompatible copied material.
- Missing meaningful 100% coverage, surviving meaningful mutants, or any
  failing required repository gate.
- Advertised feature without complete API, tests, limits, diagnostics,
  interoperability evidence where applicable, and documentation.

## Completion Criteria

`knapsack` is complete for its first stable release only when:

- pack-all, fixed-container packing, and independent verification are complete;
- all enabled geometric, weight, stock, stability, and custom constraints are
  proven for every returned plan;
- exact units and deterministic canonical results are demonstrated;
- heuristic, exact, infeasible, and budget-exhausted statuses are truthful;
- BoxPacker's relevant e-commerce capability matrix is fully classified;
- reference differential and benchmark-corpus results are reproducible;
- ordinary and adversarial workloads meet approved quality, latency, memory,
  and work budgets;
- meaningful 100% production coverage, mutation, fuzz, race, leak, security,
  vulnerability, documentation, and release gates pass from a clean checkout;
- public documentation explains the complete API, algorithms, guarantees,
  limitations, adoption, migration, and benchmark tradeoffs; and
- no unresolved high- or critical-severity correctness or security finding
  remains.
